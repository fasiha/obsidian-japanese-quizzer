# Data format reference

Quick-reference for the key data files and database tables used by the Pug app
and Node.js publish scripts. For architecture context see `CLAUDE.md`.

---

## Markdown frontmatter

All `.md` files in the repo root and subdirectories use YAML frontmatter to declare themselves and control presentation order in the iOS app's vocabulary browser.

### Required fields

**`llm-review: true`** — Declares that this file's vocabulary and grammar bullets should be processed by `prepare-publish.mjs`. Files without this field are ignored by the publish script and do not contribute to `vocab.json` or `grammar.json`.

### Optional fields

**`order: <integer>`** — Controls sort position within a directory (for content files) or the directory's position among its siblings (for `_index.md` files). Files with explicit `order` values sort before unordered files; among ordered files they sort by increasing integer value. Unordered files sort alphabetically.

#### Example

Given a directory structure:

```
Counters/
  _index.md (order: 1)
  Wago.md (order: 0)
  Counters-Must-Know.md (order: 1)
  Counters-Common.md (order: 2)
Genki 1/
  L11.md (no order)
```

In the iOS app's vocabulary browser, the sort will be:

1. `Counters/` (directory, order: 1 from `_index.md`)
   - `Counters/Wago` (order: 0)
   - `Counters/Counters-Must-Know` (order: 1)
   - `Counters/Counters-Common` (order: 2)
2. `Genki 1/` (no order, sorts after all ordered directories)
   - `Genki 1/L11` (no order, sorts alphabetically)

#### Implementation note

The `order` field is serialized into `vocab.json` as the top-level `sourceOrders` object, mapping source paths (e.g. `"Counters/Wago"`) and directory paths (e.g. `"Counters"`) to their integer sort positions. The iOS app's `VocabBrowserView` uses this map to sort directories and leaf sources with a comparison function that respects explicit order values at each path level, falling back to alphabetical order for unordered entries.

## `vocab.json`

Generated by `prepare-publish.mjs`. The iOS app fetches this file at launch and
stores it in `VocabCorpus`.

```
{
  "generatedAt": "<ISO timestamp>",
  "stories": [{ "title": "path/to/Story" }],
  "words": [ <Word entry>, … ],
  "sourceOrders": {
    "Counters": 1,
    "Counters/Wago": 0,
    "Counters/Counters-Must-Know": 1,
    "Counters/Counters-Common": 2
  }
}
```

The **`sourceOrders`** object maps source paths (e.g. `"Counters/Wago"`) and directory
paths (e.g. `"Counters"`) to explicit integer sort positions defined in Markdown
frontmatter. See the **Markdown frontmatter** section above for details.

Each **word entry**:

```
{
  "id": "1445690",                 // JMDict entry ID (string)
  "sources": ["Story A", …],       // story titles (relative path, no .md)
  "bccwjPerMillionWords": 11.61,   // BCCWJ per-million-word frequency (nullable; null if not in BCCWJ)
  "references": {                  // keyed by story title
    "Story A": [
      {
        "line": 9,                 // 1-based line number in the Markdown file
        "context": "…prose…",      // nearest preceding sentence (may be null)
        "narration": "…note…",     // non-Japanese text after the bullet tokens (may be null)
        "counter": ["くみ-クラス"], // manually annotated counter IDs for this occurrence (absent
                                   // when no `- counter:…` bullets matched this word; may contain
                                   // multiple IDs if the sentence uses the word in multiple counter
                                   // roles, e.g. both くみ-クラス and くみ-グループ)
        "llm_sense": {             // sense analysis (absent when context is empty)
          "sense_indices": [0],    // 0-based JMDict sense indices Haiku selected
          "computed_from": […],    // sorted deduplicated [context, narration] used as cache key
          "counter": ["くみ-クラス"], // counter IDs Haiku detected (empty array = asked but not a
                                   // counter; key absent = Haiku was not asked about counters for
                                   // this occurrence because ref.counter already covered it; null
                                   // = asked but Haiku returned a non-array response)
          "reasoning": "…"         // Haiku's chain-of-thought (present when freshly computed)
        },
        "annotated_forms": ["たきぎ"]  // Japanese tokens from the vocab bullet, in order
                                       // (absent for bare-ID bullets like "- 1365010")
      }
    ]
  }
}
```

Words appearing in multiple stories accumulate `sources` entries and gain one
`references` key per story. All other word data (senses, part-of-speech, etc.)
is fetched at runtime from the bundled `jmdict.sqlite`.

`annotated_forms` records the raw Japanese tokens from the annotator's vocab
bullet (e.g. `- たきぎ` → `["たきぎ"]`, `- もと 元 本` → `["もと","元","本"]`).
The iOS app resolves these against the JMDict entry's `writtenForms` to pick the
preferred display form and reading for that specific occurrence — for example,
showing 薪 with たきぎ furigana rather than the JMDict-default まき. When both a
kana and a kanji token are present and compatible, both are used; if they are
incompatible (no matching `WrittenFormGroup`), the token that appears first in
the bullet wins and a compatible partner is derived from the other slot.

## `transitive-intransitive/transitive-pairs.json`

Manually curated list of transitive/intransitive verb pairs. Each entry:

```
{
  "intransitive": { "kana": "あく", "jmdictId": "1586270", "kanji": ["開く"], "bccwjPerMillionWords": 1.38 },
  "transitive":   { "kana": "あける", "jmdictId": "1202450", "kanji": ["開ける"], "bccwjPerMillionWords": 129.26 },
  "ambiguousReason": null,          // string describing ambiguity, or null
  "examples": {
    "intransitive": "ドアが開いた。 — The door opened.",
    "transitive":   "窓を開けてください。 — Please open the window."
  },
  "drills": [                       // LLM-generated sentence pairs for quizzing
    {
      "intransitive": { "en": "…", "ja": "…", "jaFurigana": "…" },
      "transitive":   { "en": "…", "ja": "…", "jaFurigana": "…" }
    }
  ]
}
```

`bccwjPerMillionWords` is populated by `transitive-intransitive/add-bccwj-pmw.mjs` from `bccwj.sqlite`, using the same override mechanism as `vocab.json` (see `bccwj-overrides.json`). Null when the verb is absent from the BCCWJ long-unit-word list.

`jaFurigana` uses HTML `<ruby>/<rt>` markup for in-app display.

## `grammar/grammar-equivalences.json`

An array of equivalence-group objects. Each group clusters topics from Genki,
Bunpro, and/or DBJG that cover the same grammar point. Generated/extended by
the `/cluster-grammar-topics` skill; all fields except `topics` are
LLM-authored but human-reviewed.

```
[
  {
    "topics": ["bunpro:causative", "genki:causative-sentences"],
    "summary": "…one-paragraph plain-English description…",
    "subUses": ["…example use 1…", "…example use 2…", …],
    "cautions": ["…pitfall 1…", "…pitfall 2…", …],
    "stub": true     // present and true only when no annotated content sentences exist yet
  },
  …
]
```

Topic IDs are always source-prefixed: `genki:…`, `bunpro:…`, `dbjg:…`.
`prepare-publish.mjs` validates that every topic in `grammar.json` appears in
at least one group here; it also injects `summary`/`subUses`/`cautions` into
each `grammar.json` topic at publish time.

## `corpus.json`

Generated by `prepare-publish.mjs`. The iOS app fetches this file at launch and
stores it in `CorpusStore` for use in the Reader tab.

```
{
  "generatedAt": "<ISO timestamp>",
  "stories": [
    {
      "title": "path/to/Story",
      "markdown": "…full Markdown source…",
      "vocabAnnotationCounts": {
        "line_number": count,
        …
      },
      "grammarAnnotationCounts": {
        "line_number": count,
        …
      }
    },
    …
  ]
}
```

Each story includes the full Markdown source text and pre-computed counts of
vocabulary and grammar annotations per line. These counts enable the Reader tab
to display annotation badges and render collapsible panels for each line without
parsing or querying `vocab.json` or `grammar.json` at runtime.

## `Counters/counters.json`

Curated list of ~60 common counter nouns (助数詞) extracted from the
[Tofugu Japanese Counters Guide](https://www.tofugu.com/japanese/japanese-counters-guide/).
Each counter is marked "Absolutely Must Know", "Must Know", or "Common" in Tofugu's list.

```
[
  {
    "id": "こ",                      // counter ID (kana reading); unique across all entries
    "kanji": "個",                   // kanji form
    "reading": "こ",                 // kana reading
    "category": "Absolutely Must Know", // Tofugu frequency tier
    "whatItCounts": "Anything with a distinct shape",
    "countExamples": [
      "pretty much anything with a distinct shape or boundary.",
      …
    ],
    "jmdict": {
      "id": "1264740",               // JMDict entry ID
      "senseIndex": [0]              // 0-based JMDict sense index(es) for this counter meaning
    },
    "pronunciations": {              // phonetic table (numbers 1–10, plus "how-many")
      "1": { "primary": ["いっこ"], "rare": [] },
      "3": { "primary": ["さんこ"], "rare": [] },
      "6": { "primary": ["ろっこ"], "rare": [] },
      …
    }
  },
  …
]
```

The raw source data (`Counters/tofugu-350.json` and `Counters/TofuguList.tsv`)
contains all 350+ counters from Tofugu's guide and is retained for reference;
the compiled `counters.json` is a more focused pedagogical subset (top 66 counters).

## `jmdict.sqlite` — `entries` table

```sql
CREATE TABLE entries (
  id         TEXT UNIQUE,
  entry_json TEXT NOT NULL   -- JSON-encoded JMDict Word object
);
```

`entry_json` deserialises to the `Word` type from
`node_modules/jmdict-simplified-node/dist/index.d.ts`:

```typescript
interface Word {
  id: string;
  kanji: { common: boolean; text: string; tags: string[] }[];
  kana:  { common: boolean; text: string; tags: string[];
           appliesToKanji: string[] }[];
  sense: {
    partOfSpeech: string[];
    appliesToKanji: string[];   // ['*'] = applies to all kanji forms
    appliesToKana:  string[];   // ['*'] = applies to all kana forms
    // otherwise: non-empty array of specific form strings
    gloss: { lang: string; text: string; type: string | null }[];
    // … misc, field, dialect, info, related, antonym, languageSource
  }[];
}
```

`appliesToKanji` / `appliesToKana` on both `Kana` entries and `Sense` entries
use `['*']` to mean "applies to all forms"; any other value is a non-empty
array of the specific kanji or kana strings it restricts to.

Access via `jmdict-simplified-node` helpers (`findExact`, `idsToWords`, etc.)
rather than raw SQL wherever possible.

## `bccwj.sqlite` — BCCWJ word frequency tables

Generated by `.claude/scripts/build-bccwj-db.mjs` from BCCWJ TSV files. Provides per-million-word (PMW) frequency data for vocabulary.

### `bccwj` table

Main word frequency data from BCCWJ Long Unit Word (LUW) list:

```sql
CREATE TABLE bccwj (
  kanji     TEXT NOT NULL,        -- lemma (may be kana-only for kana-only words)
  reading   TEXT NOT NULL,        -- hiragana reading
  frequency INTEGER NOT NULL,     -- raw occurrence count in BCCWJ corpus
  pmw       REAL NOT NULL         -- per-million-word frequency (frequency normalized to 1M word corpus)
);
```

`kanji` is the lemma form; for kana-only words, it contains the kana text (stored in hiragana). Indexed on both `kanji` and `reading` for fast lookup.

### `bccwj_suw_counters` table

Counter noun frequency data from BCCWJ Surface Unit Word (SUW) list. Only includes words tagged as counter nouns (助数詞):

```sql
CREATE TABLE bccwj_suw_counters (
  kanji     TEXT NOT NULL,        -- lemma
  reading   TEXT NOT NULL,        -- hiragana reading
  pos       TEXT NOT NULL,        -- part-of-speech tag (接尾辞-名詞的-助数詞 or 名詞-普通名詞-助数詞可能)
  frequency INTEGER NOT NULL,     -- raw occurrence count
  pmw       REAL NOT NULL         -- per-million-word frequency
);
```

`pos` values:
- `接尾辞-名詞的-助数詞`: pure counter suffix; frequency is unambiguously counter usage
- `名詞-普通名詞-助数詞可能`: counter-capable noun; frequency includes non-counter usages

`prepare-publish.mjs` looks up the highest PMW value across all applicable kanji/reading pairs in the `bccwj` table (with fallback support for manual overrides in `bccwj-overrides.json` for words where UniDic uses a different canonical kanji than JMDict), then stores the result as `bccwjPerMillionWords` in `vocab.json`.

## `jmdict.sqlite` — `furigana` table

Populated from `JmdictFurigana.json` (the open-source JmdictFurigana project).
Primary key is `(text, reading)`.

```sql
CREATE TABLE furigana (
  text    TEXT NOT NULL,   -- written form, e.g. "怒る"
  reading TEXT NOT NULL,   -- kana reading, e.g. "おこる"
  segs    TEXT NOT NULL,   -- JSON array of FuriganaSegment
  PRIMARY KEY (text, reading)
);
```

`segs` deserialises to an array of **FuriganaSegment**:

```typescript
// A segment with "rt" is a kanji run annotated with its reading.
// A segment without "rt" (only "ruby") is a kana-only run — no annotation needed.
type FuriganaSegment =
  | { ruby: string; rt: string }   // kanji with reading
  | { ruby: string }               // plain kana
```
